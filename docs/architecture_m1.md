# Musubi M1 Architecture

Milestone 1 proves that a first-party Hermes app can securely invoke an approved local capability on a user-owned machine through Musubi.

The M1 path is:

```text
Hermes App
  -> encrypted Musubi message
Musubi Server
  -> opaque relay
Musubi CLI
  -> local policy check
Hermes Plugin
  -> local Hermes executor/runtime
Encrypted progress/result returned to app
```

## Scope

M1 includes:

- Go CLI
- local Node/Bun server for protocol development
- Cloudflare Workers + Durable Objects as hosted target
- Neon Postgres hosted database and local Postgres for dev
- JSON-RPC over stdio plugin protocol
- `echo` plugin for protocol validation
- `hermes` plugin as first real capability surface
- public-key app/device encryption envelope
- persistent message status and audit metadata
- YAML local policy with deny-by-default behavior

M1 excludes marketplaces, cloud remote plugin install, unrestricted shell, remote desktop, VPN/tunnel behavior, enterprise SSO, full metadata hiding, hardware-backed keys, WASM sandboxing, P2P mode, multi-region relay, and default offline scheduling.

## Trust Boundaries

The app owns an `app_id`, app key pair, payload encryption to device public keys, result decryption, and Musubi API calls. The app is not a plugin.

The server may see `workspace_id`, `app_id`, `device_id`, `channel`, `message_id`, status, timestamps, payload size, and audit metadata. It must not see Hermes task instructions, plugin parameters, result content, artifacts, or local file paths unless explicitly placed in visible metadata.

The CLI authenticates as a registered device, holds the device private key, decrypts device-bound payloads, checks local policy, dispatches to plugins, and encrypts app-bound results. The CLI must not contain Hermes-specific business logic.

Plugins receive decrypted payloads only after CLI policy allows execution. A plugin receives only channels it declares and is constrained by local policy permissions.

Local policy is the final execution gate:

```text
Cloud policy decides who may ask.
Local policy decides what may run.
```

## Core Services

The control plane owns users, workspaces, apps, devices, keys, grants, messages, and audit events.

The relay plane owns device WebSocket connections, online/offline state, opaque envelope delivery, acknowledgements, encrypted events, and encrypted results.

Local and hosted server modes should preserve the same protocol behavior. Hosted M1 maps `device_id` to a Durable Object `DeviceSession`.

## Key Model

Device keys are generated by the CLI during registration. The device private key stays local; the device public key is stored on the server. Device keys are used for connection challenge signing and app-to-device encryption.

App keys are generated during app creation. The app private key remains in the app runtime or dev sender environment; the app public key is stored on the server. App keys are used for device-to-app result encryption.

Each key has a stable key ID and a status: `active`, `retired`, or `revoked`.

## Message Lifecycle

M1 status transitions:

```text
created
  -> validated
  -> delivered
  -> received
  -> processing
  -> completed | failed | cancelled | expired
```

Every major transition should create an audit event. Audit metadata must not include decrypted payloads.

## Implementation Slices

1. Architecture and protocol contracts.
2. Local CLI to `echo` plugin.
3. Device identity and registration.
4. App creation and app key.
5. Grants and permission checks.
6. WebSocket relay.
7. Public-key encrypted echo end-to-end.
8. Persistent message status and audit.
9. Local policy enforcement.
10. Hermes plugin skeleton.
11. Real Hermes runtime integration.
12. Hosted Cloudflare deployment.

## Exit Criteria

M1 is complete when a user can install the Go CLI, register a device, connect it to a relay, create a first-party Hermes app, grant Hermes channels, send an encrypted Hermes task to the device public key, have the server route only ciphertext, have the CLI decrypt and policy-check locally, dispatch to Hermes, stream encrypted progress/result events back to the app, persist status/audit without plaintext, and run the same flow locally and on hosted Cloudflare infrastructure.
